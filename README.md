본 프로젝트는 **콘솔 기반 오목(FiveEye)** 및 **오셀로(Othello)** 게임을 C/C++로 구현한 학습 포트폴리오 프로젝트입니다.  
자료구조 설계, 게임 로직, 입력 처리, 보드 렌더링 등 **기본기 중심의 로직 구현 능력**을 보여주기 위해 제작되었습니다.

---

## 설계 구조 및 개발 철학 (Architecture)

본 프로젝트는 콘솔 기반 오목(FiveEye)과 오셀로(Othello)를 구현하면서, 단순한 기능 구현을 넘어서 구조적 설계를 고려하여 작성되었습니다.  
특히 아래 두 가지 원칙을 중심으로 개발을 진행했습니다.

---

### 1. 함수와 변수의 효율적 사용

반복되는 로직이나 공통적으로 사용되는 기능은 모두 함수로 분리하여 중복 코드를 최소화했습니다.  
좌표 판정, 보드 상태 변경, 승리 조건 탐색, 돌 뒤집기, 입력 유효성 검사 등은 기능별로 함수로 구성되어 있으며, 명확한 역할 기반 네이밍을 통해 가독성을 높였습니다.

예시 함수 구성:
- CheckFive() : 오목 승리 조건 판단
- FlipStones() : 오셀로 돌 뒤집기 처리
- DrawBoard() : 보드 렌더링 담당
- IsValidMove() : 좌표 유효성 검사
- InBoard() : 보드 범위 체크

이러한 구조는 유지보수와 확장성에 유리하며, 향후 AI 기능 추가 혹은 보드 크기 변경 등 구조적 변화를 쉽게 적용할 수 있습니다.

---

### 2. 로직과 렌더링의 분리

게임 규칙 처리(Logic)와 콘솔 출력(Rendering)을 명확하게 구분하여 개발했습니다.  
보드 데이터는 2차원 배열(Board[][])로만 관리하며, 출력은 별도의 함수에서 수행하도록 분리했습니다.

구조적 분리 원칙:
- 게임 상태는 Board[][] 배열이 관리
- 돌 배치, 승리 판단, 뒤집기 등은 로직 함수에서 독립적으로 처리
- 콘솔 화면 출력은 DrawBoard(), PrintStone() 등 렌더링 함수에서 전담

장점:
1. 출력 방식이 변경되더라도 게임 로직에는 영향 없음  
   (예: 콘솔 → GUI(MFC, WinAPI) 확장 가능)
2. 테스트 코드 작성이 용이함  
   (Logic 함수는 렌더링과 무관하게 단위 테스트 가능)
3. 유지보수성 향상  
   (출력과 데이터 처리를 독립적으로 관리 가능)
