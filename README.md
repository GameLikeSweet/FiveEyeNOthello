본 프로젝트는 콘솔 기반 오목(FiveEye) 및 오셀로(Othello) 게임을 C/C++로 구현한 학습용 포트폴리오 프로젝트입니다.
자료구조 설계, 입력 처리, 보드 렌더링 등 기본기 중심의 로직 구현 능력을 보여주기 위해 제작되었습니다.



## 설계 구조 및 개발 철학 (Architecture)

본 프로젝트는 단순한 기능 구현을 넘어서, 구조적 설계를 고려하여 작성되었습니다.  
특히 아래 두 가지 개발 원칙을 중심으로 구성되었습니다.



### 1. 함수와 변수의 효율적 사용

반복되거나 공통적으로 사용되는 기능은 모두 함수로 분리하여 중복 코드를 최소화했습니다.  
좌표 판정, 돌 배치, 승리 조건 검사 등의 기능은 독립된 함수로 구성되어 있으며, 역할 기반 네이밍을 통해 가독성을 높였습니다.

주요 함수 구성:
- CheckFive() : 오목 승리 조건 판단
- FlipStones() : 오셀로 돌 뒤집기 처리
- DrawBoard() : 보드 렌더링 담당
- IsValidMove() : 입력 유효성 검사
- InBoard() : 보드 범위 체크

이러한 모듈화 구조는 유지보수성과 확장성을 높이며, 향후 AI 기능 추가, 보드 크기 변경 등 구조적 수정이 요구되는 상황에서도 유연하게 대응할 수 있습니다.



### 2. 로직(Logic)과 렌더링(Render)의 분리

게임 규칙 처리와 화면 출력 기능을 명확하게 구분하여 개발했습니다.  
보드는 2차원 배열(Board[][])로만 관리되며, 렌더링은 별도의 출력 전용 함수에서 처리하도록 구성했습니다.

구조적 분리 방식:
- 게임 데이터는 Board[][] 배열이 단독으로 관리
- 승리 판단, 돌 배치, 뒤집기 등의 처리 로직은 전부 별도의 함수에서 실행
- 콘솔 출력은 DrawBoard(), PrintStone() 등 렌더링 함수에서 담당

구조 분리 장점:
1. 출력 방식이 변경되더라도 게임 로직에 영향을 주지 않음  
2. GUI 확장(예: MFC, WinAPI) 가능  
3. 테스트 코드 작성이 용이함  
4. 유지보수성 향상 (로직과 화면이 서로 독립된 구조)



요약:
본 프로젝트는 오목과 오셀로 게임 구현을 통해, 함수 단위 분리와 로직·렌더 구조 분리를 실천함으로써 유지보수성과 확장성을 고려한 프로그램 구조를 목표로 작성되었습니다.



참고: 콘솔 출력은 PC의 폰트 설정에 따라 줄 정렬에 약간의 차이가 발생할 수 있습니다.  
이는 출력 형태 차이일 뿐 로직에는 영향을 주지 않습니다.
